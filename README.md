## 구현할 기능 목록

### 열 좌표
- [X] 1~9까지의 숫자로 열 좌표를 생성하면 올바른 열 좌표를 가지고 있다.
- [X] 범위 밖의 숫자로 열 좌표를 생성하면 예외가 발생한다.

### 행 좌표
- [X] 1~10까지의 숫자로 행 좌표를 생성하면 올바른 행 좌표를 가지고 있다.
- [X] 범위 밖의 숫자로 행 좌표를 생성하면 예외가 발생한다.

### 위치
- [X] 좌표를 입력받아 위치를 생성한다.
- [X] 존재하지 않는 좌표로 위치를 생성하면 예외가 발생한다.
- [X] 좌표가 같은 위치는 동등하다.

### 기물
- [X] 팀이 맞는지 판정한다.
- [X] 자신의 이름을 반환한다.
- [X] 차, 마, 상, 장, 사, 포, 졸, 빈 구현한다.

### 보드생성
- [X] 보드에 모든 기물을 배치한다.
- [X] 보드 마상 스왑 기능 구현한다.
- [X] 초기화된 보드를 반환한다.

- [X] 보드 상태를 방어적 복사하여 반환한다.


## 이동 규칙 구현
Board.move(from, to)
→ piece.getPath(from, to) : List<Position> // 경로
→ 경로의 Piece들 추출
→ piece.canMove(List<Piece> /* 경로 */, toPiece) : void (예외)
→ 이동 실행

### 장 / 사
- [X] 직선 한 칸을 이동시키면 경로를 반환한다.
- [X] 대각선 이동시키면 예외가 발생한다.
- [X] 직선 한 칸보다 많이 이동시키면 예외를 발생한다.
- [X] 이동할 위치에 같은 팀이 있으면 예외가 발생한다.
- [X] 이동할 위치에 다른 팀이 있으면 참을 반환한다.

### 차
- [X] 한 방향으로만 이동시키면 경로를 반환한다.

- [X] 이동할 위치에 같은 팀이 있으면 예외가 발생한다.

### 포
- [X] 여러 방향으로 이동하면 예외가 발생한다.

- [X] 이동할 위치에 같은 팀이 있으면 예외가 발생한다.
- [X] 이동할 위치에 적의 포가 있으면 예외가 발생한다.

### 마

## 프로그래밍 요구사항

- Java Style Guide를 원칙으로 한다.
- indent depth를 1까지만 허용한다. (while 안에 if가 있으면 depth 2)
- 3항 연산자를 쓰지 않는다.
- else 예약어를 쓰지 않는다. switch/case도 허용하지 않는다.
- 모든 기능을 TDD로 구현해 단위 테스트가 존재해야 한다. 단, UI(System.out, System.in) 로직은 제외한다.
- 핵심 로직과 UI 로직을 분리한다. UI 로직은 InputView, OutputView 클래스로 분리한다.
- 함수(또는 메서드)의 길이가 10라인을 넘지 않도록 구현한다.
- 배열 대신 컬렉션을 사용한다.
- 모든 원시 값과 문자열을 포장한다.
- 줄여 쓰지 않는다 (축약 금지).
- 일급 컬렉션을 사용한다.
- 모든 엔티티를 작게 유지한다.
- 3개 이상의 인스턴스 변수를 가진 클래스를 쓰지 않는다.
- 도메인의 의존성을 최소한으로 구현한다.
- 한 줄에 점을 하나만 찍는다.
- getter/setter/property를 쓰지 않는다.
- 모든 객체지향 생활 체조 원칙을 지키며 구현한다.

## 팀 규칙

### 규칙 1 · 테스트 단위 기준

> 행위(Behavior)의 완결성을 기준으로 테스트를 작성한다.

**If-Then**
- `public` 메서드를 만들면 (getter 제외) → 테스트를 작성한다.
- 테스트케이스의 범위·경계값·대표값을 정의한다.
- `private` 메서드 → 별도 테스트 작성하지 않는다. (객체 자율성 보장)

**테스트 작성 형식**
```
given이고 when이면 then이다.
```

**Red / Green / Refactor**

| 단계 | 설명 |
|------|------|
| 🔴 Red | 구현 코드 없이 실패하는 테스트를 먼저 작성한다. |
| 🟢 Green | 테스트를 통과시키기 위해 가장 단순하게 구현한다. |
| ♺ Refactor | 테스트를 유지하면서 중복을 제거하고 가독성을 높인다. |

**금지**
- DB, 네트워크 등 외부 환경에 직접 연결하지 않는다.
- 입출력(I/O) 자체는 테스트하지 않는다. (순수 계산 결과만 테스트)

---

### 규칙 2 · 테스트 제외 기준

> 이미 검증된 로직은 다시 검증하지 않는다.

**If-Then**
- 이미 검증한 메서드를 다른 메서드에서 호출하는 경우 → 재검증하지 않는다.
- 단, 해당 메서드를 활용해 **새로운 로직이 생성되는 경우**는 예외로 다시 검증한다.

**테스트 포함 기준**
- 각 객체의 `public` 메서드 (getter 제외)
- 새로운 로직이 추가된 부분 (결과 판별, 점수 계산 등)
- 입력을 제외한 전체 흐름 통합 테스트 (컨트롤러 단)

**테스트 제외 기준**
- 하위에서 이미 검증된 로직
- 단순히 서비스에서 결과를 가져오는 부분
- I/O 테스트

**금지**
- 기능의 책임에서 벗어나는 케이스는 테스트하지 않는다.

---

### 규칙 3 · 테스트 어려움 대응

> 테스트가 어려우면 테스트의 대상(설계)을 리팩토링한다.

**If-Then**
- 테스트 코드를 작성하기 너무 어렵다면 → 메서드를 더 작게 분리하거나 의존성을 분리하여 설계를 즉시 개선한다.
- 도메인 로직에서 랜덤 호출이 보이면 → 바깥으로 이동하고 값으로 주입한다.
- 테스트 준비 로직이 5줄이 넘으면 → 헬퍼 메서드를 도입한다.

**리팩터링 순서**
1. 숨은 의존 제거: 내부의 랜덤·외부 의존을 식별한다.
2. 역할 분리: 해당 의존을 바깥으로 이동하고 주입받도록 설계한다.
3. 기능 분리와 역할 이관: 기능을 최소 단위로 분리한다.

**금지**
- 랜덤값 등 테스트하기 어려운 값은 객체 내부에서 생성하지 않는다. 반드시 외부에서 주입받도록 설계한다.

---

### 규칙 4 · 리팩터링 우선순위

> 컨벤션/규칙 → 역할 분리 → 세부 로직

**If-Then**
- 메서드의 매개변수가 3개 이상이면 (객체 조립·흐름 제어 제외) → 관련 변수를 새로운 객체로 묶어서 전달한다.

**리팩터링 진행 순서**

| 단계 | 설명 |
|------|------|
| 1) 기능 테스트 작성 | 리팩터링의 안전망이 될 핵심 기능 테스트를 먼저 확보한다. |
| 2) 클래스 및 메서드 분리 | 너무 많은 일을 하는 메서드를 단일 책임 단위로 쪼갠다. |
| 3) 객체 불변성 적용 | 상태 변경을 최소화하고 불변 객체를 우선 고려한다. |
| 4) 객체 구조 및 설계 변경 | 여전히 테스트가 어색하다면 그때 객체 간 관계·의존성·전체 구조를 변경한다. |
| 5) 테스트 케이스 추가 | 변경된 설계에 맞춰 필요한 테스트 케이스를 추가한다. |
| 6) 코드 정리 | 매직넘버 상수화, 중복 코드 제거, 네이밍 개선, 의미없는 주석 제거 |

**금지**
- 단순히 개수를 줄이기 위해 연관 없는 데이터를 하나의 객체에 억지로 집어넣지 않는다.

---

### 규칙 5 · 책임 분리 규칙

> 변경 이유가 다르면 분리한다. (SRP — 단일 책임 원칙)

**If-Then**
- 특정 요구사항이 변경될 때 무관한 코드까지 함께 수정해야 한다면 → 변경의 원인이 하나가 되도록 책임을 분리한다.
- 클래스를 수정해야 하는 이유가 두 가지 이상이라면 → SRP 위반으로 간주하고 즉시 별도의 클래스로 분리한다.

**책임 분리 기준 영역**

| 영역 | 예시 |
|------|------|
| 비즈니스 규칙 | 도메인 로직, 게임 규칙, 계산 로직 |
| 입력 / 출력 | View, 형식 검증 |
| 데이터 저장 | Repository, 상태 관리 |
| 외부 시스템 연동 | 랜덤, 시간, 네트워크 |
| 형식 변환 | DTO, 데이터 변환 |
| 생성 책임 | Factory, Builder |

**금지**
- 불필요하게 외부 상태를 변경하는 부수 효과(side effect)를 만들지 않는다. → setter 남용 금지

---

### 규칙 6 · 변경 범위 제한 규칙

> 단일 변경의 파급 효과가 3개 이상의 객체로 흩어지면 책임을 재분배한다.

**If-Then**
- 하나의 요구사항 변경에 3개 이상의 객체 수정이 필요하면 → 공통 로직을 추출하거나 책임을 재분배한다.
- 하나의 기능 수정에 여러 도메인 객체를 동시에 수정해야 한다면 → 캡슐화 위반·높은 결합도를 의심하고 별도의 객체로 책임을 위임한다.
- 같은 코드가 반복되면 → 공통 로직을 추출한다.
- 객체 안에 새로운 변수를 선언해야 한다면 → "원천 속성"인지 "파생 속성"인지, 라이프사이클이 일치하는지 먼저 검증한다.

---

### 규칙 7 · 테스트와 설계의 관계 규칙

> 데이터를 꺼내지 말고, 객체에게 직접 물어본다. (Tell, Don't Ask)

**If-Then**
- 데이터를 직접 꺼내어 연산하고 있다면 → 데이터를 가진 객체에게 직접 물어보도록 수정하여 변경 여파를 내부에 가둔다.
- 테스트가 깨지면 → 테스트가 아니라 설계를 수정한다.

| 구분 | 설명 |
|------|------|
| ❌ 잘못된 방식 | 외부에서 객체의 데이터를 직접 꺼내어 연산·판단을 수행한다. |
| ✅ 올바른 방식 | 데이터를 가진 객체에게 직접 물어보도록 변경한다. |
| 효과 | 변경 여파가 객체 내부에 갇혀, 외부 코드는 영향을 받지 않는다. |

---

### 규칙 8 · 상태 위치 규칙

> 서로 다른 타입이 같은 종류의 멤버 필드를 가지고 있으면 한쪽으로 책임의 이동을 고려한다.

**If-Then**
- 위치 정보는 보드가 갖는다. 기물은 자신의 이동 규칙만 안다.
- 멤버 필드(상태)의 변경 여지가 없다면 반드시 불변으로 만든다.
- 보드의 멤버 필드는 외부에서 직접 접근하지 않는다.
- 존재하지 않는 상태를 null로 표현하지 않고 별도의 객체로 표현한다. (예: `EmptyPiece`)

---

### 규칙 9 · 조건문 대체 기준

> 기물 타입에 따른 조건 분기는 다형성으로 대체한다.

**If-Then**
- 기물 타입에 따른 `switch/if`가 2곳 이상이면 → `Piece` 인터페이스를 도입하고 각 기물이 자신의 규칙을 구현한다.
- 새 기물 추가 시 기존 코드 수정이 필요하면 → 기물 등록/생성 방식을 분리한다.
- 새로운 타입을 추가할 때 기존 코드의 수정은 철저히 배제하고, 오직 새 타입에 대한 구현체만 확장하는 방식으로 변경 범위를 제한한다.

---

### 규칙 10 · 역할/인터페이스 설계 기준

> 도메인 객체들이 같은 기능, 같은 인스턴스를 2개 이상 갖는다면 인터페이스 또는 추상 클래스로 추출한다.

**If-Then**
- 기물은 자신의 이동 규칙을 알아야 한다. 다른 곳에서는 이동 규칙을 구현하지 않는다. 이동을 요청할 뿐이다.
- 도메인 객체들이 공통 기능을 가지면 → 인터페이스 또는 추상 클래스로 표현한다.

---

### 🚫 전체 금지사항

| 금지사항 | 내용 |
|----------|------|
| 불필요한 테스트 금지 | 기능의 책임 범위 밖 케이스는 테스트하지 않는다. |
| 내부 생성 금지 | 랜덤값 등 제어 불가 값은 반드시 외부에서 주입받는다. |
| setter 남용 금지 | 부수 효과를 만들지 않는다. 결과만 반환하는 구조를 우선한다. |
| 무지성 상속 금지 | 추상 클래스 / 인터페이스 / 컴포지션의 이점을 비교한 후 선택한다. |
| null 사용 금지 | 존재하지 않는 상태는 별도의 객체로 표현한다. |
